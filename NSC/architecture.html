Executive Reference Design · v1.0

A secure, two-environment AI architecture for government analytics.

One internet-connected environment for open-source collection, controlled intake, and bounded use of frontier LLMs. One air-gapped environment for protected analytics, local open-source LLMs, and decision support. Connected by a single governed, one-way transfer path.

Section 2 · Design Principles

Twelve principles that shape every decision

Principles come first — technology follows. These guide architecture, operating model, and policy in both environments.

01

Trust-boundary-first design

Separation is driven by risk and trust, not only by network topology.

02

Data sensitivity determines environment

The sensitivity class drives where data may live and who can process it.

03

Least movement of sensitive data

Only approved datasets cross environments; every movement is logged and justified.

04

Open is not the system of record

The open side is a staging, enrichment, and packaging zone — never authoritative.

05

Public LLMs are bounded tools

Used for extraction, classification, translation, summarization — never authoritative decisions.

06

Local LLMs are enterprise-anchored

Paired with retrieval, policy grounding, evaluation, and strong audit trails.

07

Human-in-the-loop for high-impact outputs

No sensitive output leaves the loop without accountable human review.

08

Deterministic, auditable transfers

Classify, DLP-scan, AV-scan, hash, sign, dual-approve, log — every artifact, every time.

09

Defense in depth

Independent controls at each layer: network, identity, data, model, application.

10

Offline-first on the closed side

Models, packages, updates arrive as signed offline bundles through the governed path.

11

Separation of duties

IT operates infrastructure; Data runs the service; Security governs policy & controls.

12

Immutable raw, versioned derivatives

Raw landing is write-once; curated and gold layers are versioned and reproducible.

Section 4 · Data Classification

Five classes govern where data can live

Every dataset, document, prompt, and model artifact carries a sensitivity label. The label determines where it can live, which LLMs may see it, and who can access it.

Class Label Allowed environments Frontier LLMs Example
C0PublicOpen ClosedAllowedNews articles, published open data
C1Internal-Open (non-sensitive)Open ClosedWith reviewNon-sensitive indicators derived from open data
C2InternalClosed onlyBlockedOperational reports, internal statistics
C3SensitiveClosed only (restricted)BlockedPersonal data, case files
C4Highly SensitiveClosed only (enclave)BlockedClassified or legally held material
!

Enforcement points

Labels are assigned at ingestion, enforced at the Policy Gate, re-enforced at the LLM Gateway, re-checked by the Transfer Gateway, and applied again at query-time on the closed side.

Section 6 · Cross-Domain Gateway

A single governed path — not a network link

The gateway is the single most security-critical component. It enforces classification, scanning, integrity, and dual-control approval before any artifact crosses the trust boundary.

What never crosses

  • Live network connections, remote shells, reverse proxies
  • Secrets, credentials, or tokens from the open side
  • User sessions, identities, or federation assertions
  • Any artifact labeled C2+ (rejected at step 1)
  • Anything missing a valid, signed manifest

Section 7 · AI / LLM Layer

Two distinct AI stacks — by design

Frontier LLMs enrich open-source material in the connected environment; local open-source LLMs produce authoritative outputs inside the air-gapped environment. Neither crosses the boundary.

☁

Open Side — Frontier LLM Layer

  • LLM Gateway routes to Gemini / Claude / GPT
  • Credentials live only in the gateway — never in user code
  • Per-team quotas & cost caps
  • Prompts & responses logged (secrets redacted)
  • Class enforcement: C0/C1 only
  • Evaluation harness gates prompt/model changes
  • Bounded tasks: extract · classify · translate · summarize
⎆

Closed Side — Local LLM Platform

  • GPU cluster for inference (vLLM / TGI / Triton)
  • Open-source models (Llama / Qwen / Mistral family)
  • Embeddings, rerankers, vector DB (pgvector / Milvus / Qdrant)
  • RAG pipeline: retrieve → rerank → filter by clearance → generate
  • Agent framework — tools restricted to internal APIs only
  • Fine-tuning (LoRA / QLoRA) on agency data; weights stay inside
  • Model registry with signed weights & provenance
Section 8 · End-to-End Flow

From collection to decision support

Every step emits metadata and audit records; lineage is preserved end-to-end.

1

Collect

Open-source data and authenticated team submissions enter the open environment.

2

Classify & route

Policy Gate labels every item and routes to LLM-eligible, prep-only, or direct-transfer.

3

Validate & standardize

Schema, quality, and policy checks; deduplication and metadata tagging.

4

Enrich with frontier LLMs

Only for C0/C1 items. Prompts & responses logged. Bounded tasks only.

5

Approve & package

Steward + security approve; package with manifest, checksums, digital signatures.

6

Transfer

Cross-domain gateway performs re-inspection and quarantines anomalies.

7

Import & integrate

Closed side verifies and integrates with internal systems of record.

8

Analyze with local AI

Local LLMs + RAG produce protected insights, clearance-aware, with citations.

9

Deliver decisions

Dashboards, reports, analyst workbench — with human sign-off on high-impact outputs.

Section 9 · Use-Case Routing

A simple rule for where each use case belongs

Use cases route by data sensitivity, need for enterprise context, and intended business outcome.

DimensionOpen EnvironmentClosed Environment
Typical data classC0 / C1C2 / C3 / C4 (and any combination)
Typical AI useExtraction, translation, summarization, classificationRAG, analyst copilots, private search, decision support
Business outcomePreparation & enrichmentIntegrated analysis & official consumption
Model typePublic LLMs via gatewayLocal open-source / self-hosted LLMs
Human reviewRequired before transferRequired for high-impact decisions
Data residencyNon-authoritative stagingAuthoritative system of record
★

Practical routing rule

If a use case needs internal protected data, combines internal and external information, or supports an official decision, it belongs in the closed environment — even if some preparation steps happened earlier in the open environment.

Section 11 · Implementation Roadmap

A phased path to production

From foundations to enterprise scale — each phase builds on the previous.

Phase 0 · 0–2 mo

Foundations

  • Governance set-up & RACI
  • IT provisions both environments
  • VLANs, IAM, secrets, monitoring
  • Cross-domain gateway installed
  • Landing zones built
Phase 1 · 2–4 mo

Open-Side MVP

  • 2–3 open-source feeds end-to-end
  • Policy Gate live (3 routing paths)
  • LLM Gateway live (one frontier model)
  • First transfers through gateway
  • Audit pipelines running
Phase 2 · 4–7 mo

Closed-Side MVP

  • Lakehouse & catalog live
  • Top internal source connectors
  • Local LLM stack live
  • Analyst Workbench pilot
  • First integrated use case
Phase 3 · 7–12 mo

Industrialize & Scale

  • Domain fine-tuned models
  • Expanded sources & use cases
  • Full observability
  • Pen test & DR drills
  • Enterprise-wide adoption
